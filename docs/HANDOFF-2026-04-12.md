### A2A Messaging — Current State

**Working:**
- `msg_send` — Send messages with `from_agent`, `message_content`, `to_agent`, `channel`, `message_type`
- `msg_read` — Read messages for an agent (polls or blocks with `wait_ms`)
- `chan_subscribe` — Subscribe to channels (`agent_identity`, `channel`)
- `chan_subscriptions` — List subscriptions

**Correct Parameter Names (Postgres backend):**
```
msg_send:        from_agent, message_content, to_agent, channel, message_type
chan_subscribe:  agent_identity, channel  (NOT "agent")
msg_read:        agent, channel, limit, wait_ms
```

**Valid message_type:** `task`, `notify`, `ack`, `error`, `event`, `text` (default)
**Valid channels:** `direct`, `team:<name>`, `broadcast`, `system`

### Known Issues

1. **Trigger broadcasts everything** — `fn_notify_new_message()` sends ALL messages to `pg_notify('new_message', ...)`. No filtering by recipient or subscription. Privacy leak.
2. **Subscriptions ignored by trigger** — `channel_subscription` table exists but trigger doesn't use it. Subscribing has no effect on push delivery.
3. **No access control** — Any agent can send to any agent. No rate limiting, no blocking.
4. **prop_create SQL bug** — Window function FILTER issue. Must insert directly via SQL.

### P199: Secure A2A Communication Model

Filed as `feature` type in `Draft` status. Research proposal covering:
- Targeted delivery (DMs only to recipient)
- Access control (role-based, rate limiting, block lists)
- Structured JSON payloads (not flat text)
- Unfriendliness detection
- Vector embeddings for semantic search
- 3 architecture options: Enhanced pg_notify, Message Queue, Hybrid
