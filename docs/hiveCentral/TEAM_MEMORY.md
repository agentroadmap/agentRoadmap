# Team Memory - AgentRoadmap

## Architecture Principles

### Data Flow
- **Postgres (Postgres)** = Source of truth for proposals
- **Markdown files** = Export/backup format for Git
- **Correct flow:** Create proposal via MCP → Postgres → Export to markdown → Git
- **⚠️ CRITICAL:** Never write markdown first! The display IDs come from Postgres, not from filenames.
  - Bad: `RFC-20260401-BUSINESS-DESIGN` (random filename-based)
  - Good: `P001`, `P002` (proper display_id from Postgres)
- **IMPORTANT:** All content lives in Postgres, markdown is just export format
- **Display IDs:** Generated by Postgres, not manually written
- **Lesson learned:** Writing files first creates random display names (e.g., RFC-20260401-BUSINESS-DESIGN)

### Git Strategy
- **origin** → gitlab.local (development, daily pushes)
- **github** → github.com (publishing, long-term sustainment)
- **Convention:** Gilbert handles git operations as Git Workflow Master (not a hard rule)

### MCP Server
- **Database:** roadmap2 (Postgres name)
- **Port:** 6421 (HTTP health endpoint)
- **Transport:** stdio (for Claude Code) + HTTP (health only)
- **Config:** `config.yml` in project root

### Config Structure
- **Init-time values:** Set during `roadmap init` (project name, database, MCP port)
- **Runtime values:** Discovered or overridden at runtime (Postgres URL, component maturity)
- **Component metadata:** Status/maturity stored in Postgres, not config

### Proposal Creation Process
- **New proposals:** Team discussion → MCP → Postgres (not file import)
- **Existing files:** Can be imported, but should be reviewed first
- **Quality check:** Verify content, maturity, and status before importing
- **Lesson learned:** Don't bulk-import without team discussion
- **Primary key:** `id` (auto-incremented by Postgres)
- **Display ID:** Auto-generated by Postgres as `P###` (0-padded), e.g., `P001`, `P002`
- **Maturity levels:** 0-3 universal scale (0=New, 1=Active, 2=Complete, 3=Mature)
- **Types:** DIRECTIVE, CAPABILITY, TECHNICAL, COMPONENT, OPS_ISSUE
- **⚠️ CRITICAL:** Display IDs are auto-generated by Postgres, not passed in by callers

### Hardcoding Prevention
- **Script:** `npm run check:hardcoded` catches hardcoded database names/URLs
- **ESLint:** Use config/env vars for external dependencies
- **Convention:** No hardcoded DB names, ports, or URLs in code

## Key Decisions (2026-04-01)

1. **Maturity model:** 0-3 universal scale across all statuses
2. **Terminology:** "proposal" not "state" (schema change)
3. **Config design:** Structured with database, MCP, git sections
4. **Git push:** Convention for Gilbert to handle, not hard rule
5. **Issue tracking:** Use MCP/Postgres proposals internally, GitHub after public release
6. **Duplicate prevention:** Check for duplicate proposals before inserting
7. **Display ID format:** Auto-generated as `P###` (0-padded), e.g., `P001`, `P002`
