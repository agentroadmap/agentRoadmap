### What Remains (The Actual Gap)

Six files in `src/apps/dashboard-web/` still import from `src/core/`:

| File | Import Source | What's Imported | Nature |
|------|--------------|-----------------|--------|
| `lib/api.ts` | `core/infrastructure/statistics.ts` | `ProposalStatistics` type | Type-only |
| `components/Statistics.tsx` | `core/infrastructure/statistics.ts` | `ProposalStatistics` type | Type-only |
| `components/ProposalList.tsx` | `core/proposal/directives.ts` | `directiveKey`, `normalizeDirectiveName`, `buildDirectiveBuckets`, `NO_DIRECTIVE_KEY` | Pure utilities |
| `components/MilestonesPage.tsx` | `core/proposal/directives.ts` | `directiveKey`, `normalizeDirectiveName`, `buildDirectiveBuckets`, `NO_DIRECTIVE_KEY` | Pure utilities |
| `utils/milestones.ts` | `core/proposal/directives.ts` | `directiveKey`, `normalizeDirectiveName`, `buildDirectiveBuckets`, `NO_DIRECTIVE_KEY` | Pure utilities (re-exports) |
| `utils/milestones.test.ts` | `core/proposal/directives` | Same as above | Test imports |

**Key insight:** These are NOT runtime Core dependencies. No filesystem I/O, no Core class instantiation, no dual data sources. They're pure TypeScript type imports and stateless utility functions — path coupling only.

### What These Imports Do

**ProposalStatistics** (from `core/infrastructure/statistics.ts`):
- Interface defining status counts, priority counts, completion percentage, health metrics
- Used in `api.ts` (fetchStatistics) and `Statistics.tsx` (rendering)
- The function `getProposalStatistics()` computes stats from proposal arrays

**Directive utilities** (from `core/proposal/directives.ts`):
- `directiveKey(name)` — lowercase trimmed key for directive comparison
- `normalizeDirectiveName(name)` — trim whitespace
- `buildDirectiveBuckets(directives, archived)` — group directives by status
- `collectArchivedDirectiveKeys()` — archived directive tracking
- `NO_DIRECTIVE_KEY` — sentinel for proposals without directives
- Used in ProposalList, MilestonesPage, milestones.ts for grouping/display

## Design Options

### Option A: Mark P301 Obsolete (Recommended)
P294 already covers the "remove filesystem Core dependency" goal. The remaining imports are type-level and utility-level — not filesystem/Postgres unification. Mark P301 `maturity: obsolete` and add a note to P294 that the cleanup is done.

**Pros:** No wasted work. P294 already did the hard part (WS migration). Remaining imports are cosmetic path coupling.
**Cons:** The import paths (`../../../core/...`) are fragile if directory structure changes.

### Option B: Narrow P301 to "Inline Core Utility Imports"
Scope P301 to just the 6 files above. Inline `ProposalStatistics` type and directive utilities into the web UI's own `types/` and `utils/` directories. This is a 30-minute task.

**Changes:**
1. Copy `ProposalStatistics` interface into `lib/api.ts` or a new `types/statistics.ts` (remove `Proposal` dependency — just use plain types matching Postgres rows)
2. Copy directive utilities into `utils/directives.ts` (they're pure functions, ~50 LOC)
3. Update 5 import paths
4. Update test import
5. `tsc --noEmit` to verify

**Pros:** Clean separation. Web UI becomes self-contained — no cross-boundary imports.
**Cons:** Duplication. If directive logic changes, web UI must be updated separately.

### Option C: Create Shared Package
Extract `ProposalStatistics` and directive utilities into a shared `@agenthive/shared` package imported by both core and web.

**Pros:** DRY. Single source of truth.
**Cons:** Over-engineered for 2 interfaces and ~5 utility functions. Introduces package management overhead.

---

## Recommendation

**Option B** — Narrow P301 to inline the remaining Core utility imports.

Rationale:
- The imports are small, stable, and self-contained (~80 LOC total)
- No risk of divergence — directive logic is display-only (UI formatting), not business logic
- Web UI becomes fully self-contained with zero cross-boundary imports
- Aligns with user's philosophy: "direct solutions over layered abstractions"
- 30-minute implementation, trivial review

If Option B is chosen, update P301's title to "Web UI: inline remaining Core utility imports" and write clear acceptance criteria.
